## 一、协议规范

### 1.1 帧格式

#### 固定帧 (Fixed Frame, 起始符 0x10)

```
┌──────────────────────────────────────────────────────────┐
│ 0x10 │ C │ A │ A │ TYPE │ CAUSE │ InfoAddr │ FCS │ 0x16 │
│ 起始符 │   │   │       │ 信息地址  │ 帧校验 │ 结束符│
└──────────────────────────────────────────────────────────┘
字节偏移:  0    1   2   3      4        6        8     9
```

| 字段 | 字节数 | 说明 |
|------|----------|------|
| 起始符 (START) | 1 | 固定为 0x10 |
| 控制域 (C) | 1 | PRM/FCB/ACD/DFC/功能码 |
| 链路地址1 (A) | 1 | 链路地址低位 |
| 链路地址2 (A) | 1 | 链路地址高位 |
| 类型标识 (TYPE) | 1 | 帧类型 |
| 传输原因 (CAUSE) | 1 | 传输原因 |
| 信息地址 (InfoAddr) | 2 | 公共地址或信息体地址 |
| 帧校验 (FCS) | 1 | 校验和 |
| 结束符 (END) | 1 | 固定为 0x16 |

---

#### 可变帧 (Variable Frame, 起始符 0x68)

```
┌─────────────────────────────────────────────────────────────────┐
│ 0x68 │ L_low │ L_high │ 0x68 │ C │ A │ A │ [ASDU] │ FCS │ 0x16 │
│ 起始符│ 长度低 │ 长度高 │ 确认符│   │   │       │ 帧校验│ 结束符│
└─────────────────────────────────────────────────────────────────┘
字节偏移:  0      1       2       3      4   5   6    7~...   N-1   N
```

**长度字段说明：**
- 标准IEC 102格式：单字节长度，值 ≤ 16
- 风光102扩展格式：双字节小端序长度 `(L_high << 8) | L_low`

**判断逻辑（实际代码实现）：**
```python
# 当 data[3] == 0x68 且 len1 == len2 且 len1 <= 16 时，使用单字节格式
# 否则使用风光102扩展的双字节小端序格式
if data[3] == 0x68 and len1 == len2 and len1 <= 16:
    self.length = len1  # 单字节
else:
    self.length = struct.unpack('<H', data[1:3])[0]  # 16-bit little endian
```

| 字段 | 字节数 | 说明 |
|------|----------|------|
| 起始符 (START) | 1 | 固定为 0x68 |
| 长度低 (L_low) | 1 | ASDU长度低位 |
| 长度高 (L_high) | 1 | ASDU长度高位（风光102扩展） |
| 确认符 | 1 | 固定为 0x68 |
| 控制域 (C) | 1 | 控制信息 |
| 链路地址1 (A) | 1 | 链路地址低位 |
| 链路地址2 (A) | 1 | 链路地址高位 |
| ASDU | 可变 | 应用服务数据单元 |
| 帧校验 (FCS) | 1 | 校验和 |
| 结束符 (END) | 1 | 固定为 0x16 |

### 1.2 控制域 (Control Field)

```
┌────────────────────────────────────────────────┐
│  bit7 │ bit6 │ bit5 │ bit4 │ bit3 │ bit2 │ bit1 │ bit0 │
│ 保留   │ PRM   │ FCB   │ ACD/DCM│ DFC │  FC2  │  FC1  │  FC0  │
└────────────────────────────────────────────────┘
```

| 位 | 名称 | 说明 | 值含义 |
|----|------|------|----------|
| bit7 | 保留 | 保留位，未使用 | - |
| bit6 | PRM (启动/从动) | 0 = 从动站（响应），1 = 主动站（请求） | 0/1 |
| bit5 | FCB (帧计数位) | 0 = FCB=0，1 = FCB=1 | 检测帧丢失 |
| bit4 | ACD/DCM | PRM=0时：ACD（数据可请求）<br>PRM=1时：DCM（方向控制） | 0/1 |
| bit3 | DFC (数据流控制) | 0 = 数据流正常，1 = 数据流受限制 | 0/1 |
| bit2-0 | FC2-FC0 (功能码) | 定义请求/响应类型 | 0-7 |

## 二、ASDU 结构

### 2.1 标准ASDU结构

```
┌────────────────────────────────────────────────────────────────────────┐
│ TYPE │ VSQ  │ COT  │ DevAddr │ RecAddr │ [信息体]    │
│ 类型标识│ 可变结构│ 传输原因│ 设备地址│ 记录地址│ 信息对象列表   │
└────────────────────────────────────────────────────────────────────────┘
字节偏移:  0     1        2        4~5     6        7~...
```

| 字段 | 字节数 | 说明 |
|------|----------|------|
| 类型标识 (TYPE) | 1 | ASDU类型 |
| VSQ | 1 | 可变结构限定词 |
| 传输原因 (COT) | 1 | 传输原因 |
| 设备地址 (DevAddr) | 2 | 设备地址 |
| 记录地址 (RecAddr) | 1 | 记录地址 |
| 信息体 | 可变 | 信息对象列表 |

### 2.2 VSQ (可变结构限定词)

```
┌────────────────────────────────┐
│ bit7 │ bit6-0 │
│ SQ   │ NUM    │
│ 序列/单个│ 信息元素数 │
└────────────────────────────────┘
```

| 位 | 说明 |
|----|------|
| bit7 (SQ) | 0 = 单个信息对象，1 = 顺序信息对象序列 |
| bit6-0 (NUM) | 信息元素数量 (0-127) |

### 2.3 风光102规约文件传输ASDU结构

```
┌────────────────────────────────────────────────────────────────────────────────────┐
│ TYPE │ VSQ  │ COT  │ DevAddr │ RecAddr │   文件名(64字节)    │ 文件内容... │
│ 类型标识│ 可变结构│ 传输原因│ 设备地址│ 记录地址│ 0x00填充的GBK编码文件名   │ 实际文件数据  │
└────────────────────────────────────────────────────────────────────────────────────┘
字节偏移:  0     1        2        4~5     6        7~70               71~...
```

| 字段 | 字节数 | 说明 |
|------|----------|------|
| 类型标识 (TYPE) | 1 | 文件传输类型(0x90, 0x143-0x149等) |
| VSQ | 1 | 可变结构限定词 |
| 传输原因 (COT) | 1 | 文件传输状态(0x07结束/0x08中间) |
| 设备地址 (DevAddr) | 2 | 设备地址 |
| 记录地址 (RecAddr) | 1 | 记录地址 |
| 文件名 | 64 | GBK编码的文件名，空余用0x00填充 |
| 文件内容 | 可变 | 实际文件数据字节 |

### 3.2 TCP流重组

**流键格式：**
```python
conn_key = (src_ip, src_port, dst_ip)  # 三元组
rev_key = (dst_ip, dst_port, src_ip)   # 反向键
```

**特点：**
- 使用源三元组（不包含目的端口）
- 同一TCP连接的双向数据流共享同一个ID
- 反向键用于复用连接ID

**重组过程：**
```
TCP包1: payload = "10 C A A TYPE..."
    ↓
添加到流: tcp_streams[conn_key] += payload

TCP包2: payload = "CAUSE InfoAddr FCS 16"
    ↓
添加到流: tcp_streams[conn_key] += payload

TCP流 = "10 C A A TYPE...CAUSE InfoAddr FCS 16"
    ↓
扫描流中IEC 102帧起始符(0x10/0x68/0xE5)
    ↓
提取完整帧: IEC102Frame(tcp_stream)
```

### 3.3 方向判断规则（基于端口）

```python
def get_dir_by_port(src_port: int, dst_port: int) -> int:
    """根据端口判断DIR方向
    规则：端口3000或3001为终端，另一端为主站
    """
    TERMINAL_PORTS = {3000, 3001}
    src_is_terminal = src_port in TERMINAL_PORTS
    dst_is_terminal = dst_port in TERMINAL_PORTS

    if src_is_terminal and not dst_is_terminal:
        return 1  # 终端→主站 (↑)
    elif dst_is_terminal and not src_is_terminal:
        return 0  # 主站→终端 (↓)
    else:
        return 0  # 默认主站→终端
```

| 场景 | 源端口 | 目的端口 | 方向 | 符号 |
|------|----------|-----------|------|------|
| 终端→主站 | 3000/3001 | 其他 | DIR=1 | ↑ |
| 主站→终端 | 其他 | 3000/3001 | DIR=0 | ↓ |
| 同类型端口 | 3000 | 3001 | 无法判断 | ↔ |

```python
if asdu_data[0] in (0x90, 0x96, 0x97, 0x98, 0x99, 0x9A) or \
   (0x143 <= asdu_data[0] <= 0x149):
    if len(asdu_data) >= 70:  # 6字节头 + 64字节文件名
        name_bytes = asdu_data[6:70]
        # 过滤0x00截断符
        filtered = bytes([b for b in name_bytes if b != 0])
        if filtered:
            # 优先使用GBK解码（中文文件名）
            try:
                file_name = filtered.decode('gbk', errors='ignore').strip()
            except:
                # 降级使用ASCII解码
                file_name = filtered.decode('ascii', errors='ignore').strip()
```

### 5.2 文件传输流程

```
主站请求文件
    ↓
子站发送文件数据（多帧）
    ├─→ 帧1: Cause=0x08 (中间帧), data=部分文件内容
    ├─→ 帧2: Cause=0x08 (中间帧), data=部分文件内容
    ├─→ 帧3: Cause=0x08 (中间帧), data=部分文件内容
    └─→ 帧4: Cause=0x07 (最后一帧), data=剩余文件内容
    ↓
主站确认接收
```

### 5.3 文件合并规则

```python
def merge_file_data(frames):
    """合并多个帧的文件数据"""
    all_data = []

    for frame in frames:
        asdu = frame['iec']['asdu']
        file_data = asdu['file_data']  # 字节数组
        cause = asdu['cause']

        if file_data:
            all_data.extend(file_data)

    return bytes(all_data)
```

| 传输原因 | 处理方式 |
|----------|----------|
| 0x08 (中间帧) | 累加文件数据 |
| 0x07 (最后一帧) | 累加文件数据（传输结束） |

## 十一、常见问题

### Q1: 为什么需要TCP流重组？
A: IEC 102帧可能被TCP分片传输，单个TCP包可能包含帧的一部分、完整的一帧或多个完整帧。需要重组TCP流数据后再解析IEC 102帧。

### Q2: 如何判断文件传输是否完成？
A: 检查传输原因（Cause）：
- Cause=0x07：文件最后一帧，传输结束
- Cause=0x08：文件未传输结束（中间帧）

### Q3: 文件名如何解码？
A: 优先使用GBK编码（支持中文文件名），失败后降级使用ASCII编码。文件名固定64字节，用0x00填充，提取时过滤掉填充符。

### Q4: 如何区分终端和主站？
A: 基于端口号判断：
- 端口3000或3001为终端
- 另一端为主站
- 从终端端口发出 → DIR=1（终端→主站）
- 发往终端端口 → DIR=0（主站→终端）
