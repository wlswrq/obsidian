## 一、现状分析

### 1.1 问题：大量重复代码

**35个Handler类，几乎完全相同，每个文件约175行，只有2处不同：**

| 差异点 | 说明 |
|--------|------|
| Redis Key 常量 | `alert:pv_agc_rate_alert_` / `alert:svg_availability_rate_` 等 |
| `getInvokeTargetAlertType()` | 返回不同的枚举值 |

**重复内容：**
- `handle()` 主流程（60%相同）
- `processLastMonthNonExaminedAlert()`（10%）
- `processNonExaminedAlert()`（10%）
- `processExaminedAlert()`（10%）
- Redis 操作方法（10%）

### 1.2 代码量统计

```
35 个 Handler × ~175 行 = ~6125 行
去除2处差异后，实际唯一逻辑 ≈ 50-80 行
```

## 二、优化方案对比

### 方案A：模板方法模式（继承）

**思路：** 在 `hi5000-platform/hi5000-alert` 中创建通用抽象模板类，子类只需实现2个方法。

```java
// 公共模板类
public abstract class AbstractCommonIndicatorsAlert extends AbstractIndicatorsAlert {

    @Override
    public final List<TBusinessAlert> handle(...) {
        // 统一的处理流程，子类不可重写
    }

    // ========== 子类需实现 ==========
    protected abstract String getRedisKeyPrefix();
    public abstract InvokeTargetAlertTypeEnum getInvokeTargetAlertType();
}

// 简化后的子类
@Component
public class PvAgcOperationalRateAlert extends AbstractCommonIndicatorsAlert {
    @Override
    protected String getRedisKeyPrefix() {
        return "alert:pv_agc_rate_alert_";
    }
    @Override
    public InvokeTargetAlertTypeEnum getInvokeTargetAlertType() {
        return InvokeTargetAlertTypeEnum.PV_AGC_OPERATIONAL_RATE;
    }
}
```

| 维度 | 评估 |
|------|------|
| 代码量 | 新增模板基类 ~500行，子类 ~25行 |
| 优点 | 复用彻底，父类控制完整流程 |
| 缺点 | 紧耦合，继承层次深 |

---

### 方案B：组合模式（推荐）

**思路：** 创建公共处理类 `IndicatorsAlertHelper`，各Handler注入使用。

```java
// 公共处理类
@Component
public class IndicatorsAlertHelper {
    
    @Resource
    private RedisUtil redisUtil;

    public List<TBusinessAlert> process(TBusinessAlertConfig config,
                                        TBusinessIndicators indicators,
                                        String redisKeyPrefix) {
        // 统一处理流程
    }
}

// 简化后的Handler
@Component
public class PvAgcOperationalRateAlert extends AbstractIndicatorsAlert {

    @Resource
    private IndicatorsAlertHelper alertHelper;

    private static final String REDIS_KEY_PREFIX = "alert:pv_agc_rate_alert_";

    @Override
    public List<TBusinessAlert> handle(...) {
        return alertHelper.process(config, indicators, REDIS_KEY_PREFIX);
    }
}
```

| 维度 | 评估 |
|------|------|
| 代码量 | 新增Helper类 ~200行，Handler ~40行 |
| 优点 | 松耦合，可单独测试Helper，扩展灵活 |
| 缺点 | Handler需注入Helper |

---

### 方案C：配置驱动（彻底优化）

**思路：** 废除每指标一个类，改为单一Handler + 数据库配置。

```
现状：35个类 → 优化后：1个类 + 配置
```

**实现：**

```java
// 单一日志告警处理Handler
@Component
public class IndicatorsAlertHandler extends AbstractIndicatorsAlert {

    @Resource
    private RedisUtil redisUtil;

    @Override
    public List<TBusinessAlert> handle(TBusinessAlertConfig config,
                                      TBusinessIndicators indicators, ...) {
        // 从配置获取Redis Key前缀
        String redisKeyPrefix = config.getRedisKeyPrefix();
        if (redisKeyPrefix == null) return null;
        
        // 统一处理逻辑
        // ...
    }
}
```

**配置表数据示例：**

| id | alertClass | dataId | redisKeyPrefix |
|----|-----------|--------|----------------|
| 1 | ALERT_INDICATOR | 101 | alert:fault_recording_alert_ |
| 2 | ALERT_INDICATOR | 102 | alert:capacity_qualification_rate_alert_ |
| 3 | ALERT_INDICATOR | 103 | alert:pv_agc_rate_alert_ |

**新增指标告警：只需插入配置，无需编写新类。**

| 维度 | 评估 |
|------|------|
| 代码量 | 1个Handler ~250行 |
| 优点 | 代码量最少，扩展最方便（配置驱动） |
| 缺点 | 特殊逻辑需扩展机制支持 |

## 三、方案对比总结

| 维度 | 方案A继承 | 方案B组合 | 方案C配置驱动 |
|------|-----------|-----------|---------------|
| 代码新增 | ~500行 | ~200行 | ~250行 |
| Handler代码 | ~25行/个 | ~40行/个 | 0（合并为1个） |
| 总类数量 | 36个 | 36个 | 2个 |
| 耦合度 | 紧耦合 | 松耦合 | 最松 |
| 扩展方式 | 继承+重写 | 注入+传参 | 配置+新 Handler |
| 新增指标 | 写新类 | 写新类 | 只加配置 |
| 测试难度 | 中等 | 最易 | 易 |

## 四、最终推荐

### 推荐：**方案B + 方案C 结合**

1. **近期（方案B）：**
   - 新增 `IndicatorsAlertHelper` 公共处理类
   - 各Handler简化为40行，注入Helper
   - 代码量减少 ~70%

2. **远期（方案C）：**
   - 扩展 `TBusinessAlertConfig` 表，增加 `redisKeyPrefix` 字段
   - 迁移为配置驱动
   - 新增指标告警只需配置，无需写类

### 架构演进图

```
【现状】                          【近期-方案B】                    【远期-方案C】

┌─────────────────┐           ┌─────────────────┐            ┌─────────────────┐
│ PvAgcHandler    │ ←相同逻辑→│ IndicatorsAlertHelper │       │ IndicatorsAlert │
│ SvgHandler      │           │  (统一处理流程)   │            │  (配置驱动)      │
│ ...             │           └─────────────────┘            └─────────────────┘
└─────────────────┘                   ↑                                   ↑
      ↓                             │                                   │
   175行重复代码 ──────────────→ 注入使用，40行 ──────────────→ 新增只需配置
```
